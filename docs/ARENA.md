# Arena — same-data multi-variant ablation harness

This document is written so another AI (or person) can run and extend `scripts/arena.py`
without re-deriving the design. Read it fully before editing.

## What it does (one line)

Train several config variants on the **exact same self-play data**, then pit them
**head-to-head** and rank by win rate — so differences come from the *config*, not from
luck in data sampling or from one variant training longer.

## Why this exists (the problem it fixes)

Comparing two separately-trained runs is noisy: each generates its own data, and small
differences are swamped by data-sampling variance and unequal training. (We learned this the
hard way: a "separate slot embedding" arm looked +4.6% in one head-to-head but was a dead tie
over 10 runs, because the +4.6% was just extra training, not the architecture.)

Arena removes both confounds:
- **Same data**: one rollout per update is shared by all variants → identical experience.
- **Same training amount**: all variants do the same number of updates on the same batches.

So the only thing that differs is each variant's config (loss weights / feature flags / arch).

## How it works (mechanism)

```
per update:
  variant 0 (the "generator") collects ONE self-play rollout on CPU  (SubprocVecEnv)
  variant 0 trains on it (on-policy, normal PPO)
  variants 1..N each train on the SAME rollout, each on its own GPU, with its own config
after N updates:
  round-robin head-to-head among all variants  ->  win-rate matrix + overall ranking
```

- Each variant is an independent `DaVinciCodePolicy` on its own `cuda:i` (no DDP, no gradient
  sync across GPUs — so none of the NCCL/`oras`-style hangs we hit before).
- The shared rollout (obs/actions/rewards, GAE advantages) is computed once by variant 0's
  `PPOTrainer` and reused; each variant recomputes its *own* old-log-probs/old-values for its
  PPO ratio (`variant_update` in the script).

### The one honest caveat — off-policy

The data is generated by **variant 0's** policy. Variants 1..N therefore train on data that is
*not* from their own policy (off-policy). This is fine for **short, warm-started** comparisons
because all variants start from the same checkpoint and stay close for a while. It degrades if
you run very long (policies diverge). Two consequences:

1. Keep runs bounded (tens–hundreds of updates), warm-started from a shared checkpoint.
2. **`mean_reward` cannot rank variants** — they all see the same data, so it's identical for
   all. The **head-to-head win rate is the only valid discriminator.** This matches what we
   already proved: head-to-head is ground truth, `mean_reward` can mislead.

## Run it

```bash
export LD_LIBRARY_PATH=/home/temp/py310/lib   # this box only; skip if your env is fine
export PYTHONPATH=$PWD
python scripts/arena.py \
    --checkpoint checkpoints/best_model_control.pt \   # warm-start ALL variants from here (recommended)
    --updates 200 \
    --n-envs 4800 \          # shared rollout size per update (CPU-bound; set workers to cores)
    --workers <cpu_cores> \
    --h2h-episodes 2000      # games per head-to-head pairing at the end
```

- **GPU mapping is automatic**: variant `i` goes to `cuda:(i % num_gpus)`. With 4 GPUs and 4
  variants, one per GPU. With 1 GPU, all share it (fine — model is tiny).
- Outputs: each variant saved to `checkpoints_arena/<name>.pt`, plus the win-rate matrix and
  overall ranking printed to stdout.

## Define your variants (the part you edit)

Edit the `CONFIGS` list at the top of `scripts/arena.py`. **Variant 0 is the data generator**;
put your baseline first. Each variant is a name + a dict of `PPOConfig`-field overrides:

```python
CONFIGS = [
    {"name": "baseline",     "overrides": {}},
    {"name": "no_belief",    "overrides": {"belief_coef": 0.0}},
    {"name": "high_entropy", "overrides": {"ent_coef": 0.03}},
    {"name": "low_clip",     "overrides": {"clip_range": 0.1}},
]
```

Any field of `PPOConfig` (see `src/trainer.py`) can be overridden: `belief_coef`, `ent_coef`,
`vf_coef`, `clip_range`, `clip_range_vf`, `gamma`, `gae_lambda`, `learning_rate`, `n_epochs`,
`batch_size`, etc.

## Comparing a NEW feature or loss (how to extend)

The arena compares *configs*; a brand-new feature must first be made toggleable. Three steps:

1. **Add a flag to `PPOConfig`** (`src/trainer.py`), e.g. `use_rope: bool = False`.
2. **Use that flag** where the behavior lives:
   - model architecture change → read it in `src/model.py` (e.g. gate RoPE in the encoder),
     and make sure `DaVinciCodePolicy`/`ObservationEncoder` accept and thread the flag;
   - loss change → read it in the loss computation. NOTE: arena's per-variant loss is in
     `variant_update()` in `scripts/arena.py`, **not** `PPOTrainer.update()`. If your loss
     change must apply to the non-generator variants, implement it in `variant_update()` too
     (variant 0 uses `PPOTrainer.update()`; variants 1..N use `variant_update()` — keep them
     in sync for a fair test).
3. **Add a variant** toggling it: `{"name": "rope", "overrides": {"use_rope": True}}`.

If the feature changes the model constructor signature, also update the two model-build sites
in `scripts/arena.py` (`gen` via `PPOTrainer`, and the variant loop's `DaVinciCodePolicy(...)`).

## Files involved

- `scripts/arena.py` — the harness (data generation reuse, `variant_update`, round-robin h2h).
- `src/trainer.py` — `PPOConfig` (the override surface) + `PPOTrainer` (variant 0's collect+update).
- `src/model.py` — `DaVinciCodePolicy` (per-variant model).
- `src/buffer.py` — rollout buffer + GAE (shared advantages come from here via variant 0).
- (head-to-head is built into `arena.py` as the `h2h()` function — no separate eval script needed.)

## Gotchas / limitations (read before trusting results)

- **Off-policy** (above): keep runs bounded; trust head-to-head, not `mean_reward`.
- **`PPOTrainer.load` on main is broken for PyTorch 2.6** (`weights_only=True` default rejects
  the numpy-scalar config). Arena works around it with its own loader (`load_into`,
  `weights_only=False`). If you call the trainer's `load` elsewhere, add `weights_only=False`.
- **CPU-bound**: the shared rollout collection dominates wall-clock; set `--workers` to the
  CPU core count. More GPUs do NOT speed up data generation.
- **Shared advantages**: all variants use variant 0's GAE advantages (computed from variant 0's
  value baseline) as the target signal, but each uses its own old-log-probs/old-values for the
  PPO ratio and value clip. This is a deliberate simplification for fairness; it's an
  approximation, and the head-to-head is the final arbiter.
- **Determinism**: head-to-head uses a fixed RNG seed for side assignment; env deals vary per
  run, so re-running gives slightly different win rates (±a couple % at 2000 games). Run a few
  times or raise `--h2h-episodes` if a margin is near the noise band.
